1. Resumen Ejecutivo

Se realizó una evaluación técnica de seguridad sobre una aplicación Python desarrollada deliberadamente con vulnerabilidades críticas para fines académicos y de laboratorio.

El objetivo de la auditoría fue identificar debilidades de seguridad asociadas a malas prácticas de programación, exposición de información sensible y uso inseguro de funciones del sistema.

Durante el análisis se identificaron vulnerabilidades críticas relacionadas con:

  • SQL Injection
  • Command Injection
  • Uso inseguro de eval() y exec()
  • Deserialización insegura
  • Cross-Site Scripting (XSS)
  • Criptografía débil
  • Exposición de secretos hardcodeados
  • Almacenamiento inseguro de datos sensibles
  • Uso inseguro de subprocess y shell=True
  • Falta de validación de entradas
Resultado General:
La aplicación presenta un nivel de riesgo CRÍTICO, permitiendo potencialmente ejecución remota de código, acceso no autorizado a información sensible, manipulación de bases de datos y compromiso total del sistema.

12 CRÍTICA Deserialización Insegura con marshal CWE-502 9.1 Carga insegura de objetos binarios controlados por usuario. marshal.loads(eval(datos)) Corrupción de memoria y ejecución arbitraria. Evitar marshal con entradas externas.

14 ALTA Cross-Site Scripting (XSS) CWE-79 8.2 Construcción de HTML sin sanitización de entradas. html = "<h1>Hola " + nombre + "</h1>" Robo de sesiones y ejecución de scripts maliciosos. Escapar contenido HTML y sanitizar entradas.
15 ALTA Server-Side Template Injection (SSTI) CWE-1336 8.8 Uso inseguro de plantillas HTML controladas por usuario. render_template_string(html) Ejecución de expresiones y acceso a información sensible. Evitar renderizar entradas del usuario como plantillas.
16 ALTA Bypass de sanitización con safe CWE-79 8.0 Uso del filtro safe permitiendo HTML arbitrario. plantilla = "{{ comentario|safe }}" Inyección de scripts maliciosos persistentes. Eliminar safe y aplicar escaping automático.
17 ALTA Criptografía Débil MD5 CWE-327 7.5 Uso de MD5 para almacenamiento de contraseñas. hashlib.md5(password.encode()).hexdigest() Cracking rápido de hashes mediante diccionarios. Usar Argon2, bcrypt o PBKDF2.
18 ALTA Criptografía Débil SHA1 CWE-327 7.4 Uso de SHA1 para firmas criptográficas. hashlib.sha1((dato + SECRET_KEY).encode()).hexdigest() Riesgo de colisiones y falsificación de firmas. Usar SHA-256 o HMAC seguro.
19 ALTA Generación Insegura de Tokens CWE-330 8.1 Uso de random() y random.randint() para sesiones. token = str(random.randint(1000, 9999)) + usuario Predicción y secuestro de sesiones. Usar secrets.token_hex() o secrets.token_urlsafe().
20 ALTA Almacenamiento de contraseñas en texto plano CWE-256 8.6 Contraseñas almacenadas sin cifrado ni hashing seguro. usuarios.append({"password": password}) Exposición completa de credenciales. Aplicar hashing fuerte y sal aleatoria.
21 ALTA Exposición de información sensible en logs CWE-532 8.0 Registro de contraseñas y secretos en archivos de log. logging.warning("Login con password: %s", password) Filtración de credenciales y secretos internos. No registrar datos sensibles.
22 ALTA Exposición de información sensible por consola CWE-200 7.8 Impresión de credenciales y tokens directamente en salida estándar. print("Contraseña administrador:", ADMIN_PASSWORD) Divulgación de secretos críticos. Eliminar impresión de información sensible.
23 ALTA Path Traversal CWE-22 8.6 Acceso inseguro a archivos mediante rutas controladas por usuario. contenido = open(ruta, "r").read() Lectura de archivos arbitrarios del sistema. Validar y normalizar rutas.
24 ALTA Subida Insegura de Archivos CWE-434 8.8 Archivos guardados utilizando nombres controlados por usuario. destino = "uploads/" + archivo.filename Subida de malware y sobrescritura de archivos. Validar extensiones y renombrar archivos.
25 ALTA Lectura de archivos sin restricciones CWE-400 7.7 Lectura completa de archivos sin límites de tamaño. datos = open(ruta, "rb").read() Consumo excesivo de memoria y DoS. Implementar límites y validaciones.
26 ALTA Falta de Autenticación en Funciones Críticas CWE-306 8.7 Funciones sensibles pueden ejecutarse sin autenticación previa. def actualizar_password_sql(): def eliminar_usuario_sql(): Usuarios no autorizados pueden modificar o eliminar cuentas. Implementar autenticación y control de acceso.
27 ALTA Gestión Insegura de Sesiones CWE-384 8.1 Tokens de sesión predecibles y sin expiración segura. sesion = str(random.random()) + usuario Secuestro y fijación de sesiones. Usar tokens criptográficamente seguros y expiración.
28 ALTA Almacenamiento de Información Sensible en Texto Plano CWE-312 8.0 Datos sensibles almacenados directamente sin cifrado. INSERT INTO usuarios VALUES(NULL, ..., password, ...) Exposición masiva de credenciales y datos personales. Aplicar cifrado y hashing seguro.
29 ALTA Server-Side Request Forgery (SSRF) CWE-918 8.6 Solicitudes HTTP realizadas hacia URLs controladas por el usuario. requests.get(url) Acceso a servicios internos y exfiltración de datos. Restringir destinos y validar URLs.
30 ALTA Uso Inseguro de Markup() CWE-79 8.0 Se marca contenido HTML no sanitizado como seguro. return Markup(pagina) Ejecución persistente de scripts maliciosos. No utilizar Markup() sobre contenido dinámico.
31 MEDIA Uso inseguro de YAML CWE-20 6.5 Uso de yaml.load() sin SafeLoader. data = yaml.load(texto) Posible ejecución de objetos arbitrarios. Utilizar yaml.safe_load().
32 MEDIA Falta de timeout en requests HTTP CWE-400 5.9 Solicitudes HTTP realizadas sin timeout configurado. requests.get(url) requests.post(url, data=payload) Bloqueos y agotamiento de recursos. Agregar timeout y manejo de excepciones.

41 BAJA Uso de Variables Globales Sensibles CWE-562 3.9 Datos críticos almacenados en variables globales compartidas. usuarios = [] sesiones = {} pagos = [] Incrementa superficie de ataque y errores de estado. Reducir uso de estado global compartido.

44 BAJA Ausencia de Auditoría de Seguridad CWE-778 3.5 No existe trazabilidad segura de eventos críticos. logging.info(...) logging.warning(...) Dificulta investigaciones forenses y monitoreo. Implementar auditoría centralizada y segura.
45 BAJA Diseño Inseguro de Arquitectura CWE-657 3.9 Mezcla lógica de negocio, seguridad y acceso a sistema en un solo módulo. menu() main() funciones críticas compartidas Aumenta complejidad y probabilidad de errores de seguridad. Separar responsabilidades mediante capas seguras.

9. Recomendaciones Generales

  1. Aplicar validación y sanitización de entradas.
  2. Eliminar completamente el uso de eval(), exec() y shell=True.
  3. Utilizar consultas parametrizadas.
  4. Aplicar hashing moderno con bcrypt o Argon2.
  5. Utilizar variables de entorno para secretos.
  6. Implementar análisis SAST con herramientas como Bandit y SonarQube.
  7. Implementar revisiones de código seguras.
  8. Adoptar prácticas OWASP Top 10 y SDL.
  9. Implementar manejo seguro de archivos.
  10. Aplicar políticas de control de dependencias.

10. Conclusión

La aplicación evaluada presenta múltiples vulnerabilidades críticas derivadas de malas prácticas de desarrollo seguro y ausencia de controles básicos de ciberseguridad.

Se identificaron debilidades graves relacionadas con ejecución arbitraria de código, exposición de secretos, SQL Injection, deserialización insegura y manejo inseguro de datos sensibles.

El nivel de riesgo general se considera CRÍTICO, recomendándose remediación inmediata de las vulnerabilidades más severas y adopción de metodologías de desarrollo seguro basadas en OWASP, SAMM y SDL.
