Informe de Evaluación de Ciberseguridad

Aplicación Python Vulnerable

Evaluación basada en buenas prácticas de desarrollo seguro, metodología SAMM, SDL y lineamientos OWASP.

1. Resumen Ejecutivo

Se realizó una evaluación de seguridad sobre una aplicación Python diseñada deliberadamente con múltiples vulnerabilidades críticas. El objetivo fue identificar riesgos que afectan la confidencialidad, integridad y disponibilidad del sistema.

Durante la revisión se detectaron vulnerabilidades severas como:

  • SQL Injection
  • Ejecución arbitraria de código
  • Exposición de credenciales y secretos
  • Uso inseguro de criptografía
  • Deserialización insegura
  • Cross-Site Scripting (XSS)
  • Command Injection
  • Falta de validación y sanitización de entradas

Estas vulnerabilidades podrían permitir acceso no autorizado, robo de información sensible, ejecución remota de comandos, escalamiento de privilegios y compromiso total del sistema.

2. Metodología Utilizada

La evaluación se realizó considerando principios de:

  • OWASP Top 10
  • SAMM (Software Assurance Maturity Model)
  • SDL (Security Development Lifecycle)
  • Buenas prácticas de desarrollo seguro en Python

Se analizaron controles relacionados con:

  • Validación y sanitización de entradas
  • Protección de datos sensibles
  • Gestión segura de autenticación
  • Uso seguro de librerías y funciones
  • Prevención de inyecciones
  • Criptografía segura
  • Manejo seguro de archivos
  • Pruebas y verificación de seguridad

# Severidad Vulnerabilidad Descripción Código Vulnerable Recomendación
1 CRÍTICA SQL Injection Concatenación directa de entradas del usuario en consultas SQL. Permite manipulación de base de datos. consulta = "SELECT * FROM usuarios WHERE nombre LIKE '%" + texto + "%'" Utilizar consultas parametrizadas con placeholders.
2 CRÍTICA Ejecución Arbitraria con eval() El uso de eval() permite ejecutar código malicioso. resultado = eval(expresion) Reemplazar por lógica controlada, parsing seguro o ast.literal_eval().
3 CRÍTICA Ejecución Arbitraria con exec() Permite ejecutar instrucciones Python ingresadas por el usuario. exec(comando_python) Eliminar completamente el uso de exec().
4 CRÍTICA Command Injection Entrada del usuario concatenada en comandos del sistema. os.system("ping " + host) Validar entradas y usar listas de argumentos sin shell=True.
5 ALTA Subprocess con shell=True Riesgo de ejecución de comandos arbitrarios. subprocess.run(comando, shell=True) Evitar shell=True y sanitizar parámetros.
6 CRÍTICA Credenciales Hardcodeadas Claves y secretos almacenados directamente en el código fuente. ADMIN_PASSWORD = "SuperPassword123" Usar variables de entorno o vaults seguros.
7 ALTA Exposición de Información Sensible Contraseñas y tokens impresos en consola y logs. print("Token activo:", JWT_TOKEN) Nunca exponer secretos en logs o pantallas.
8 ALTA Criptografía Débil (MD5 y SHA1) Uso de algoritmos inseguros para hashing y firmas. hashlib.md5()
hashlib.sha1() 		Utilizar bcrypt, Argon2 o SHA-256 con salt.
9 ALTA Cifrado Casero XOR Implementación criptográfica insegura y fácilmente reversible. encrypt_xor_casero() Utilizar librerías estándar como cryptography.
10 CRÍTICA Deserialización Insegura Uso inseguro de pickle y marshal con datos del usuario. pickle.loads(eval(datos)) Nunca deserializar datos no confiables.
11 ALTA Cross-Site Scripting (XSS) HTML generado sin sanitización. html = "<h1>Hola " + nombre + "</h1>" Escapar entradas y evitar renderizado inseguro.
12 ALTA Uso Inseguro de YAML yaml.load() puede ejecutar objetos arbitrarios. yaml.load(texto) Utilizar yaml.safe_load().
13 MEDIA Requests sin Timeout Riesgo de bloqueo y denegación de servicio. requests.get(url) Definir timeout y validar URLs.
14 ALTA Subida Insegura de Archivos No se validan nombres ni tipos de archivo. archivo.save(destino) Validar extensión, tamaño y sanitizar nombres.
15 ALTA Directory Traversal Posibilidad de acceder archivos arbitrarios. ruta = "data/" + nombre Validar rutas y restringir accesos.
16 MEDIA Lectura de Archivos sin Límite Riesgo de consumo excesivo de memoria. open(ruta, "rb").read() Limitar tamaño y validar archivos.
17 ALTA Almacenamiento Inseguro de Tarjetas Datos financieros almacenados en texto plano. pagos.append({"tarjeta": tarjeta}) Cumplir PCI-DSS y cifrar información sensible.
18 MEDIA Tokens Inseguros Uso de random() para sesiones y tokens. random.random() Utilizar secrets.token_hex().
19 ALTA Contraseñas en Texto Plano Passwords almacenadas sin hashing seguro. password TEXT Aplicar hashing fuerte con bcrypt o Argon2.
20 MEDIA Denegación de Servicio Bucle infinito sin control de recursos. while True: Implementar límites y control de memoria.

4. Riesgos para la Organización

  • Acceso no autorizado a bases de datos
  • Robo de credenciales y tokens
  • Compromiso total del servidor
  • Exposición de información financiera
  • Pérdida de disponibilidad del sistema
  • Incumplimiento normativo y legal
  • Daño reputacional

5. Recomendaciones Generales

  1. Implementar validación y sanitización de entradas en cliente y servidor.
  2. Eliminar completamente el uso de eval(), exec(), pickle.loads() y comandos inseguros.
  3. Migrar a consultas parametrizadas.
  4. Utilizar autenticación segura y hashing moderno con bcrypt o Argon2.
  5. Proteger secretos mediante variables de entorno.
  6. Aplicar principios OWASP Top 10.
  7. Integrar herramientas automáticas de análisis como: Bandit, SonarQube y linters de seguridad.
  8. Implementar revisiones de código y pruebas de seguridad continuas.
  9. Aplicar metodología SDL y SAMM en el ciclo de desarrollo.
  10. Incorporar políticas de gestión segura de dependencias.

6. Conclusión

El sistema evaluado presenta múltiples vulnerabilidades críticas que comprometen seriamente la seguridad de la aplicación y la protección de los datos procesados.

La ausencia de validaciones, el uso de funciones peligrosas, la exposición de secretos y la implementación insegura de mecanismos criptográficos representan un alto riesgo operativo.

Se recomienda priorizar la remediación inmediata de las vulnerabilidades críticas y adoptar un enfoque de desarrollo seguro basado en estándares reconocidos como OWASP, SAMM y SDL.
